A failed dbt build buries the real problem under a wall of errors and skips. dbt-debugger cuts through it, explains why, and maps the blast radius across your whole pipeline.

It uses dbt's own semantics, not a guess

  • →status=error means the node ran and its parents were fine, so it's a root cause.
  • →status=skipped means something upstream failed, so it's a casualty, attributed to its nearest failed ancestor.
  • →In dbt build, a failing test gates the models below it, so its model is treated as the blocking root cause.
  • →Tests, source freshness, and the stale threshold roll up per model. Expand a failing test to copy its compiled query and pull the bad rows.
